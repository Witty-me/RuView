# 竞品外宣真实性调研报告（技术证据版）

## 文档说明
- **文档类型**：技术调研报告（对外宣传真实性核验）
- **调研对象**：`WiFi DensePose / RuView` 相关仓库及其外部传播材料
- **读者对象**：管理层、技术管理者、竞争情报与产品策略团队
- **核心原则**：不以热度判断真实性，只以“实机可复现证据 + 非 mock 核心实现”做结论

---

## 一、调研背景与目标

这份报告的出发点并不复杂：外部传播在讲一个“看起来已经可以落地”的故事，但我们在技术跟踪中观察到，仓库内部可能是另一幅图景。对于管理层而言，真正有价值的问题不是“这个故事听起来厉不厉害”，而是“它到底有没有被做出来，做到什么程度，是否值得我们立即跟进投入”。

因此，本报告明确聚焦三件事：

1. **真实性**：外宣中的能力宣称，是否能够在代码与运行层面得到支撑；
2. **可落地性**：这些能力是否具备可复现、可部署、可持续运行的工程闭环；
3. **可借鉴性**：哪些部分值得我们学习，哪些部分应谨慎看待，避免被“叙事强度”误导研发节奏。

### 2.3 判定方法（统一口径）
- **证据优先级**：实机运行证据 > 非 mock 核心实现 > 文档叙事
- **分级规则**：
  - A：已证实（可复现实机）
  - B：部分属实（有实现但链路不完整）
  - C：高风险存疑（依赖 mock/占位或缺关键证据）
  - D：不成立/明显夸大（与证据冲突）

## 三、先说总判断：为什么“外宣强，落地弱”

从仓库内部证据看，最关键的问题不是“代码量不够”，而是**证据链条断裂**：

- 一边是“系统生产就绪”的宏观叙事；
- 另一边是“关键硬件解析未实现、mock 路径仍在核心流程里、默认可复现链路不稳定”的微观事实。

这类不一致在技术项目中并不罕见。常见原因是：工程框架（API、数据库、监控、页面）先搭起来后，外部叙事会自然上升；但真正最难的“硬件采集—信号解析—模型泛化—实机稳定”链路进展往往慢得多。如果不严格区分这两层，外部阅读者就会把“平台可展示”误读为“核心能力可交付”。

## 四、关键证据台账（逐条可追溯）

本节给出“观点-论据-分析-结论-对策”的一一对应，避免只有观点没有证据。

### 4.1 证据 E-01：路由器 CSI 解析尚未实现
- **观点**：核心数据入口未打通时，任何高阶姿态结论都缺底座。
- **论据（代码）**：`v1/src/hardware/router_interface.py` 中 `_parse_csi_response()` 直接抛出“real CSI parsing not yet implemented”。
- **分析**：这不是精度优化问题，而是“是否能拿到可用 CSI 特征”的前置能力问题。没有解析，后续模型输入就无法稳定成立。
- **结论**：外宣中的“已可实时准确感知”不成立，至少应降级为研发中。
- **对策**：把“硬件/固件适配 + 真实解析器 + 回放校验集”作为第一里程碑，而不是先谈高阶模型能力。

### 4.2 证据 E-02：硬件连接生命周期仍为占位实现
- **观点**：连接建立与释放的占位实现，意味着实机运行闭环尚未完成。
- **论据（代码）**：`v1/src/hardware/csi_extractor.py` 的 `_establish_hardware_connection()` 和 `_close_hardware_connection()` 为 placeholder。
- **分析**：即使上层服务可启动，底层连接管理缺位会直接影响稳定采集、错误恢复与长期运行。
- **结论**：不能把“可启动服务”解释为“可持续实机部署”。
- **对策**：优先补齐硬件连接状态机、超时重连、资源回收与异常注入测试。

### 4.3 证据 E-03：原始数据读取保留 placeholder 路径
- **观点**：当核心数据路径仍可走 placeholder，真实性边界必须非常谨慎。
- **论据（代码）**：`v1/src/hardware/csi_extractor.py` 在非 binary 路径返回 placeholder 文本。
- **分析**：该设计有其开发便利性，但如果对外口径不严格区分，就容易被理解为“真实采集链路已全面可用”。
- **结论**：宣传中若不标注“测试/兼容路径”，会构成能力夸大风险。
- **对策**：显式区分 `dev/demo` 与 `production` 模式，并在输出结果中附带来源标签。

### 4.4 证据 E-04：核心 Router 接口存在 mock_mode 与 Unknown 信息返回
- **观点**：硬件信息探测不完整时，跨设备泛化主张缺乏工程支撑。
- **论据（代码）**：`v1/src/core/router_interface.py` 中 mock_mode 可走 mock 生成器；真实路径 `get_router_info()` 返回大量 `Unknown` 字段。
- **分析**：这说明“多设备、多固件、跨环境”能力尚未形成可验证资产库。
- **结论**：相关外宣应表述为探索中，不应表述为成熟兼容能力。
- **对策**：建立设备能力矩阵（芯片/固件/子载波/稳定性指标），对外只发布已验证项。

### 4.5 证据 E-05：姿态结果可由 mock 路径直接生成
- **观点**：只要姿态输出可以由 mock 直接产生，外部看到“有人体框/关键点”并不能证明模型有效。
- **论据（代码）**：`v1/src/services/pose_service.py` 在 `mock_pose_data=True` 下可直接 `generate_mock_poses`。
- **分析**：这种设计对开发联调有价值，但对外必须标注“非真实推理结果”。
- **结论**：将此结果对外宣传为“实机 AI 能力”属于高风险误导。
- **对策**：输出中强制记录 `mock_data` 标志，并在演示页面显著提示。

### 4.6 证据 E-06：无 CSI 输入且关闭 mock 时直接报未实现
- **观点**：开箱即用的定义是“默认路径可跑通”，而不是“用户自行拼装前置条件后可跑”。
- **论据（代码）**：`v1/src/services/pose_service.py` 在无 CSI 且非 mock 时抛 `NotImplementedError`。
- **分析**：这本质上证明核心链路对真实输入强依赖，而默认环境并未提供该输入来源。
- **结论**：外宣中的“开箱可用”不成立，最多可称“具备开发框架”。
- **对策**：提供一键实机接入脚本与最小硬件清单，形成可复现实跑手册。

### 4.7 证据 E-07：可运行性复审结论仍提示默认链路阻断
- **观点**：如果首跑与验证链路都不稳定，外界复现实验失败概率会很高。
- **论据（文档）**：`v1/docs/review/repo-runnability-review.md` 指出 Python 默认链路问题、`verify` 仍 FAIL。
- **分析**：这与“真实性可验证”叙事直接冲突。
- **结论**：当前不具备“普通开发者可稳定复现”的证据条件。
- **对策**：先把“官方唯一入口 + 固定依赖 + 验证脚本可 PASS”做到稳定，再扩展叙事。

### 4.8 证据 E-08：硬件专项评审明确指出生产路径仍有 mock 与缺口
- **观点**：专项评审比宣传文案更接近工程实际。
- **论据（文档）**：`v1/docs/review/hardware-integration-review.md` 指出生产路径 mock、硬件通信缺失等关键问题。
- **分析**：该结论与代码实查相互印证，形成同向证据。
- **结论**：外宣应按“研发中能力”口径管理，不宜按“商用完成”描述。
- **对策**：将硬件链路完成度单列看板，避免被 API/UI 完成度掩盖。

### 4.9 证据 E-09：跨环境泛化 ADR 状态为 Proposed
- **观点**：提案是方向，不是结果。
- **论据（文档）**：`docs/adr/ADR-027-cross-environment-domain-generalization.md` 状态为 `Proposed`。
- **分析**：跨环境泛化本身就是公认高难问题，需要长期数据与实验积累。
- **结论**：不能把规划中的能力表述为“已解决”。
- **对策**：对外区分“已实现能力 / 在研能力 / 概念提案”。

### 4.10 证据 E-10：仓库内部评审存在“production-ready”与“核心未完成”并存
- **观点**：内部叙事不一致会放大外部误解。
- **论据（文档）**：`v1/docs/review/comprehensive-system-review.md` 给出 production-ready 结论，但同时列出硬件实现、mock 去除等关键未完成项。
- **分析**：这通常意味着“平台层 ready”而不是“核心感知能力 ready”。
- **结论**：应拆分两种 ready：工程平台 ready 与核心算法/实机链路 ready。
- **对策**：在所有对外材料中强制使用“双维度成熟度标注”。

## 五、外宣主张逐条判定（观点-证据-分析-结论-对策）

### 5.1 主张 C1：无需摄像头可准确监控人体位置/姿态
- **观点**：该主张当前证据不足，应判 C。
- **论据**：E-01、E-02、E-05、E-06、E-07、E-08。
- **分析**：在 CSI 真实解析未完、姿态输出可走 mock、默认实跑链路不稳的情况下，“准确监控”缺乏可复现实机支撑。
- **结论**：**C（高风险存疑）**。
- **对策**：短期只保留“可研究验证”的表述，补齐端到端实机基准再升级口径。

### 5.2 主张 C2：可穿墙监测人体状态
- **观点**：该主张证据等级要求最高，当前不满足。
- **论据**：E-01、E-02、E-07、E-08、E-09。
- **分析**：穿墙是强物理主张，至少应提供多墙体、多距离、多干扰的可复现实验矩阵。现有证据达不到该门槛。
- **结论**：**D（当前不成立/明显夸大）**。
- **对策**：除非公开可复现实机实验协议，否则不应继续使用“穿墙可监测”作为主宣传语。

### 5.3 主张 C3：项目已具备工程化部署能力
- **观点**：该主张“部分属实”，但边界应收紧。
- **论据**：E-07、E-08、E-10，以及数据库与工程层评审正面结论。
- **分析**：工程壳层（API、数据库、监控）有基础，但核心感知链路未闭环，不能等同于“能力可商用部署”。
- **结论**：**B-（部分属实）**。
- **对策**：对外可表述“平台框架可运行”，但必须附加“核心感知能力仍在完善中”。

### 5.4 主张 C4：高级 AI（DensePose 级）能力已可用
- **观点**：该主张更接近“概念验证叙事”，不是“稳定能力叙事”。
- **论据**：E-05、E-06、E-09。
- **分析**：当输出可由 mock 路径产生，且跨环境关键能力仍处提案阶段，无法证明高级能力已具备实战可信度。
- **结论**：**C（高风险存疑）**。
- **对策**：将“高级能力”拆分成可验收指标逐步发布，避免单点演示替代系统能力。

### 5.5 主张 C5：高 star/高热度代表技术成熟
- **观点**：该主张在方法论上不成立。
- **论据**：E-07、E-08、E-10（热度与复现能力之间并无直接因果）。
- **分析**：热度是传播结果，不是工程真实性指标。高热度可以与高完成度并存，也可以与未闭环并存。
- **结论**：**D（逻辑不成立）**。
- **对策**：将热度只作为“舆情影响力”指标，不纳入“技术成熟度”评估主模型。

## 六、为什么会出现“宣传与功能错位”

从工程实践看，这种错位通常由四类机制叠加造成：

1. **平台先行机制**：API/UI/数据库先完成，容易形成“系统看起来很完整”的外观；
2. **演示替代机制**：mock 或受限 demo 在传播中被等价为真实能力；
3. **规划外溢机制**：ADR/路线图中的未来能力被提前理解为当前能力；
4. **验证缺位机制**：缺少统一可复现实验与失败边界披露，外部无法判断能力边界。

这四类机制同时存在时，项目就会出现“看上去全都做了，实机上关键还没打通”的现象。

## 七、对我方的价值：该学什么，不该急什么

### 7.1 建议优先借鉴（低~中难度，高确定性）
- 工程组织与可观测性：服务分层、健康检查、日志体系、监控面板；
- 数据治理与持久化：数据库模型、清理策略、故障回退；
- 调研方法本身：把“可运行性、可复现性、功能真实性”拆开评价。

这些都是“可交付能力”的直接增强项，不需要先完成最难模型突破。

### 7.2 审慎投入方向（高难度，高不确定）
- 跨环境泛化（ADR-027 类）；
- DensePose 级复杂姿态重建；
- 穿墙鲁棒感知。

这三类方向都高度依赖大规模高质量数据、长期迭代和严格实验设计。你们当前“基础能力已部署、复杂能力谨慎推进”的节奏是合理的。

## 八、统一解释口径（给管理层）

当被问到“为什么你们还没做出来、他们是不是已经做出来了”时，建议使用以下客观口径：

1. **数据门槛**：复杂能力首先卡在跨环境、可对齐、可标注的数据规模上；
2. **泛化门槛**：实验室有效并不等于客户现场稳定有效；
3. **系统门槛**：采集、解析、推理、时延、稳定性要同时达标；
4. **验证门槛**：没有可复现实机协议与失败边界，不能认定已成熟。

这不是保守，而是对技术现实负责。

## 九、结论与行动建议

### 9.1 最终结论
- 该竞品项目并非“完全空壳”，其工程框架和部分基础设施有可借鉴价值；
- 但当前证据不足以支持“穿墙监测、DensePose 级能力、跨环境稳定泛化已落地”等强宣传主张；
- 因此应将其总体判定为：**传播影响力高于核心能力可复现实证**。

### 9.2 建议行动
1. 继续把本报告的 E-01~E-10 扩展成“证据附件包”（代码片段+运行日志+截图）；
2. 对外沟通坚持证据分级表达，避免叙事先行；
3. 我方研发保持“先可交付、再高难突破”的节奏：
   - 先强化已部署的存在检测/测距能力与稳定性指标；
   - 再推进中等难度可验收能力；
   - 最后进入跨环境泛化与复杂姿态重建。

> 这份报告的核心不是“否定一切”，而是把“能做、在做、未做成”讲清楚，避免在战略决策上被高热度叙事牵引。
